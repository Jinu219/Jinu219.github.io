From Simulation Code to a Research Workflow: PySDM Seeding Lab v1.0.0

Build the Lab
PySDM
reproducible research
scientific scope
How PR #5 connected targeted response qualification, resumable execution, Arrow caching, observation-mapping audits, and a machine-checked scientific claim contract.
Author

Jinu

Published

2026-07-21

The previous development log separated absolute rain-state qualification from seeding-response qualification, then connected common seeds, process isolation, detached jobs, and bounded parallel execution. Experiments could finally survive a long night on the lab server. What remained open was a project-level contract: How much computation is enough, how should model transitions be compared with observations, and what can v1.0 actually claim?

PySDM-Seeding-Lab PR #5, at its 21 July 2026 checkpoint, answers those questions across 16 commits and 71 changed files, with 7,054 additions and 98 deletions. This was not a changelog-sized feature bundle. It was an effort to place execution planning, numerical qualification, observation-mapping review, reporting, and scientific limitations in one evidence chain.

Core decision. PySDM Seeding Lab v1.0.0 does not demonstrate that cloud seeding works. It completes a reproducible research workflow that runs experiments, qualifies numerical behavior, audits observation mappings, reports results, and preserves the boundary of every scientific claim.

Turning an open-ended project into one bounded question

PR #4 left six practical and scientific problems unresolved:

  • Would the seeding-minus-control response converge at a finer reference resolution?
  • Could an interrupted qualification safely resume without rerunning valid members?
  • Could the Streamlit dashboard avoid repeatedly parsing large CSV files?
  • What contract would align a model spectrum transition with an observed drizzle event?
  • How should real observations be retained when they do not map cleanly to parcel time?
  • Where should an expanding development program stop and call itself v1.0?

These are all reproducibility questions. Reproducing a result requires more than a successful number: the reason for running a calculation, the rejected interpretation, and the unusable evidence must survive beside it. The sixteen commits in PR #5 therefore move from a finer targeted run to a machine-checked scientific claim boundary.

Asking the high-resolution question in 40 runs, not 70

The original rain_response_standard plan called for a 2.5 s timestep, 1,600 seeded and 1,600 background super-droplets, and five common random seeds. Running its complete plan would require 70 control/seeding executions. Pilot evidence had already shown that absolute state and response behaved very differently, so executing every combination was not automatically the best scientific question.

The new targeted qualification plan retains only the finest reference and each numerical axis’s next-finest comparison:

4 one-factor-at-reference cases
× 5 common random seeds
× control / seeding
= 40 physical PySDM executions

That is 42.9% less work than the 70-execution plan without removing a comparison used by the 5% convergence decision. A dry run freezes the plan first; the confirmed qualification then runs serially with one worker so its resource commitment remains explicit.

Long runs can still be interrupted. Resume does not trust a directory merely because its name exists. A member is reusable only when its normalized execution configuration has the same SHA-256 fingerprint and its configuration, summary, and CSV are readable and successful. Missing, corrupt, failed, and configuration-mismatched members are rerun. Aggregation, evidence, and reports are rebuilt, while each attempt records reuse and rerun counts in qualification_plan.json.

Doubling repeated-read speed without replacing CSV

Repeated dashboard views made large CSV parsing increasingly expensive. Replacing CSV with a binary format would also replace the human-readable provenance, migration, and download contract.

The columnar-cache design keeps CSV as the canonical source of truth and stores Arrow IPC in a hidden, disposable cache. A missing, corrupt, or stale cache is rebuilt from CSV. Systems without pyarrow, and tables below the default size threshold, continue to read CSV. Arrow is a rebuildable performance layer, not a scientific result.

The benchmark repeatedly read a real 101 × 505 result with 51,005 cells:

Format Raw CSV Cold build/read Warm cache Warm / CSV
Parquet prototype 24.981 ms 370.801 ms 33.058 ms 0.756×
Arrow IPC v2 16.271 ms 61.597 ms 8.076 ms 2.015×

Warm Arrow reduced latency by about 50.4%, reached break-even after roughly seven total reads, and occupied 1.2467 times the CSV storage. Parquet was not selected merely because it is familiar: it was slower than CSV for this result shape. Exact dataframe equality and measured behavior—not format preference—made the decision.

The Parquet screening and Arrow IPC confirmation were separate sequential runs, so the meaningful comparison is each format’s ratio to its own CSV baseline rather than a direct comparison of absolute timings.

Separating observation ingestion from observational validation

Before comparing the model’s internal 1% rain-liquid transition with observed drizzle onset, the project needed to define what one observation event means. The new observation-event contract requires event and case identity, onset and uncertainty, model-time offset, time origin, source ID, observation method, event definition, sampling context, and mapping status.

An immutable comparison package records every threshold candidate, onset error, RMSE, within-uncertainty fraction, source hashes, and a manifest. The lowest-MAE candidate is never promoted automatically to a calibration. Synthetic rows test only software behavior; even real observations remain spatiotemporal_proxy when sampled volume and parcel time are not defensibly aligned.

BASTALIAS: real data from a moving volume

The EUREC4A BASTALIAS importer extracts persistent drizzle detections from approximately 1.5 s airborne radar–lidar NetCDF data. It retains the source file’s full SHA-256 and sensitivity across the -15, -17, and -20 dBZ cloud–drizzle boundaries. All 400 samples in the reviewed ten-minute window passed the time quality flag; the selected -20 dBZ event appeared 7.121 s after the window began.

The aircraft, however, observes different horizontal volumes as it moves. A persistent echo is not necessarily the temporal history of one parcel. The importer therefore forces mapping_status = spatiotemporal_proxy even though the evidence class is observation. The pipeline compared the event against 27 model candidates, but produced zero direct-temporal rows. The mapping audit explicitly prevents those rankings from calibrating the operational 1% floor.

ARM ENA: a fixed column is still not a parcel history

The ARM ENA pipeline selects fixed-column KAZR/KAZRARSCL data as the next candidate. Credentials are accepted only through environment variables; partial downloads are atomically replaced; source hashes, QC fields, height selection, and -20/-17/-15 dBZ sensitivity are retained in the package.

A zenith radar is closer to a time series than a moving aircraft, but it observes an Eulerian column through which wind advects different parcels. Radar reflectivity is also not the model’s rain-liquid fraction. No ARM credential or real source file was available at this checkpoint, so a synthetic ARM-format NetCDF verified software behavior only. The candidate audit therefore closes the v1.0 investigation with a proxy disposition, not a successful external calibration.

The finer run found a boundary, not an effect

The targeted-run evidence completed all 20 case-seed pairs and all 40 physical executions, with zero failed cases and zero failed members.

Qualification family Within 5% Maximum error v1.0 decision
Absolute control/seeding rain state 60 / 60 4.180% Supported for the tested profile
Seeding-minus-control response 11 / 105 2465.158% Quantitative response unsupported

All five finest-reference seeds had a positive final rain-response direction, with a mean final difference of about 0.0001377 kg/kg. That 5/5 result is neither a confidence interval nor a significance test, and it is not a converged effect estimate. Every seed independently rejected response support. Maximum response errors along the timestep, seeded super-droplet count, and background super-droplet count axes were 1630.411%, 2272.468%, and 2465.158%, respectively.

The turning point was refusing to rename a desirable sign as an effect. Absolute rain state is supported only for the reviewed collision-ON marine profile, its four OFAT cases, five common seeds, and finest-to-next-finest comparisons. The smaller difference between two states did not converge with resolution. Rather than launch all 70 executions in hope of a pass, the workflow preserves direction and spread as descriptive results and convergence as a separate rejected claim.

Managing scientific claims with a schema

A limitation written only in a paragraph is easy for the next analysis or UI to forget. PR #5 makes release/v1_scientific_scope.json the machine-readable source of truth for claim order, status, scope, limitations, and evidence.

Claim Status Permitted interpretation
Absolute rain state Supported Reviewed collision-ON marine profile and resolution gate
Positive response direction Descriptive only Five positive finest-reference directions
1% rain-liquid fraction Operational only Internal transition exploration with sensitivity audits
Quantitative enhancement Unsupported Response convergence did not support it
External calibration Unsupported Both datasets remain parcel-time proxies
Field efficacy Unsupported No matched field intervention evidence

Tests now prevent “observation ingested” from silently becoming “model validated,” or “positive sign” from becoming “effect established.” Preserving evidence that cannot support a claim is part of reproducibility too.

Giving v1.0 a finite finish line

There is always another useful task in an expanding research codebase. The v1.0 release checklist defines five dependency-ordered gates:

  1. a reproducible simulation and analysis platform;
  2. real BASTALIAS observation ingestion and a mapping audit;
  3. a conservative external-validation disposition;
  4. a frozen scientific scope review; and
  5. a versioned, documented, CI-checked release candidate.

All five gates are complete. The local fast-regression suite passed 77 tests. In the specified GitHub Actions run, Windows fast regression, Ubuntu fast regression, and real PySDM integration on Ubuntu all passed. CI also checks the release manifest and scientific-scope contract.

This does not mean the PR has already been merged into main or that the tag has been published. Under release/v1.0.0.json, PR #5 is paused at the mandatory pre-merge Build the Lab checkpoint. Publishing this evidence-backed account comes before the develop → main merge, release-CI confirmation, and v1.0.0 tag.

Not avoiding failure, but making it interpretable

Completing the v1.0.0 release candidate does not mean every scientific question produced a positive answer. It means the workflow can reproduce where the answer stops.

  • Absolute rain state is numerically supported within a limited tested profile.
  • Response direction is descriptive, while quantitative response is unsupported.
  • The 1% floor is an operational definition, not a universal drizzle-onset standard.
  • BASTALIAS and ARM ENA are valuable observation proxies, not Lagrangian parcel calibrations.
  • Surface rainfall gain, causal effect, and real-world cloud-seeding efficacy are outside v1.0’s claims.

The project did not fail because the desired convergence did not appear. It learned how to decide, reproducibly, what the evidence permits us to say. Simulation code becomes a research workflow when the scope of each number survives alongside the number itself.

Questions deliberately left for v1.1 and later research

The following work remains outside the finite v1.0 gate:

  • measuring real-PySDM server scaling for serial, four-worker, and eight-worker execution;
  • direct Lagrangian observation mapping with parcel trajectories and residence times;
  • additional datasets for transition-threshold and cadence review; and
  • stronger convergence and matched-observation evidence before any quantitative seeding-response claim.

v1.0.0 does not pretend to have answered these questions. It leaves an executable contract for what the next experiment must establish—and what language remains off limits until it does.
